# openyak 0.1.0 发布说明（草案）

## 版本定位

`0.1.0` 是当前 Rust 主实现的首个公开版本草案。这个版本的定位不是“历史实现归档查看器”，而是一套可以从源码直接构建、运行并验证的本地 coding-agent CLI。

本文已在 `2026-04-08` 结合最新一轮本地全量验收和 fresh release-binary 58 步命令 rerun 同步刷新。

一句话概括：

> `0.1.0` 标志着 openyak 已经进入 Rust-first、可运行、可维护、可继续扩展的阶段。

## 本版本包含的核心能力

- 交互式 REPL
- 单次 prompt 执行
- 工作区级工具系统
- 会话保存、恢复、导出
- 插件发现与管理
- 本地 agents / skills 发现
- attach-first、本地-only 的 TypeScript / Python SDK alpha（基于当前 `openyak server` `/v1/threads` 合约）
- 命名 `toolProfiles` ceiling 与受限 hidden browser tools（`BrowserObserve` / `BrowserInteract`）
- `openyak foundations [task|team|cron|lsp|mcp]` / `/foundations [task|team|cron|lsp|mcp]` 只读发现入口
- MCP、OAuth、registry-backed parity foundation/operator tools，以及 registry-backed LSP operator surface + `openyak server` 本地 thread/session surface
- Git / GitHub 工作流命令

## 版本亮点

### 1. Rust 主实现正式成形

当前版本中，`rust/` 已经是仓库的主产品面，而不是实验性替代品。`openyak` 二进制具备完整的主工作流，包括：

- 交互式会话
- 单次任务执行
- 配置加载
- 权限模式
- 工具调用
- 会话管理

### 2. 插件、Hook 与工具系统已接通

`0.1.0` 不再是“只有命令框架”的版本。当前已经具备：

- 插件加载、安装、启用、禁用、卸载、更新
- 插件工具聚合
- 插件 hook 与基础 hook 合并执行
- Shell、文件编辑、搜索、Web、todo、skill、agent 等内置工具

同时，插件 manifest 的文件路径已经做了边界加固：

- manifest 中声明的文件路径必须相对插件根目录解析
- 解析后的最终路径必须保持在插件根目录内
- 不能借由非字面量路径逃逸到插件目录之外

### 3. 平台路径与配置规则统一

用户目录、配置目录与 skills 发现规则已经统一为共享实现：

- `OPENYAK_CONFIG_HOME` 优先决定用户级 `.openyak` 目录
- `CODEX_HOME` 次优先决定用户级 `.codex` 目录，并参与用户根目录推导
- 否则回退到平台默认用户目录
- 不再把当前工作目录的 `./.openyak` 当作默认用户目录

skills 同时支持：

- `skills/<name>/SKILL.md`
- `skills/.system/<name>/SKILL.md`

### 4. OAuth 站点改为显式配置

当前版本的 `openyak login` 不再依赖任何内置默认站点：

- 必须由 `settings.oauth` 显式提供 `clientId`、`authorizeUrl`、`tokenUrl`
- OAuth 后端由使用者自己提供，CLI 不会回退到任何默认服务 URL
- 配置了 `manualRedirectUrl` 时，CLI 会进入手动粘贴回调 URL 或 query string 的模式
- OAuth token 优先写入系统凭据库；只有系统凭据库不可用时才回退到用户级 `credentials.json`
- 仓库提供了两份可直接填写的模板：
  - `rust/docs/oauth.settings.loopback.template.json`
  - `rust/docs/oauth.settings.manual-redirect.template.json`

### 5. `openyak server` 的本地服务边界已明确

当前版本将 `server` crate 以 `openyak server` 的形式正式 surfaced 为本地 HTTP/SSE thread server：

- 暴露 `/v1/threads`
- 保留 legacy `/sessions` compatibility routes
- 把线程状态持久化到工作区 `.openyak/state.sqlite3`
- 启动时打印实际绑定地址

同时这条边界也被明确收紧：

- 它是本地 thread/session server，不是 codex-style full app-server
- SDK 公共边界仍然锁定在 `/v1/threads`，不会把 `/sessions` 作为新公共 contract

### 6. 运行时日期不再硬编码

系统提示和状态输出中的当前日期由运行时按本地时钟生成，只在测试里注入固定值，避免长期漂移的常量污染“今天/最近/最新”相关判断。

### 7. Git 本地 `.openyak` 状态不再阻塞提交工作流

`/commit` 和 `/commit-push-pr` 现在会正确跳过 `.openyak/settings.local.json`、`.openyak/sessions/` 等本地文件，不再因为这些工作区本地状态导致 `git add` 失败。

### 8. Windows 链路完成一轮收口

当前版本已经针对 Windows 做过一轮关键兼容性收口，覆盖：

- 构建与 `clippy`
- 路径与环境变量回退
- skills 扫描
- 外部命令解析
- 浏览器启动与 `gh` 调用相关链路

### 9. 编译产物命令面验收进一步收口

最近一轮可执行文件验收补上了几类高频 CLI 细节：

- 顶层子命令 `--help` 现在返回对应命令的专属帮助，而不是误执行命令本体
- `/diff` 现在会显示未跟踪文件，同时继续排除 `.openyak/settings.local.json`、`.openyak/sessions/`、`.omx` 等本地状态噪音
- 同一工作区内若有多个 `openyak server` 实例交叠运行，thread discovery 文件现在会按 `pid` 做 owner-safe 清理，较早退出的实例不会误删较新实例的发现入口
- 最近这轮命令面巡检已经收敛为自动化二进制 smoke/regression 覆盖，并在 `2026-04-08` 的 fresh release-binary rerun 中再次验证了逐个顶层命令 help、direct slash CLI、resume-safe slash command 链路，以及 `login` / `onboard` / `prompt` / self-target `package-release` 的受控失败路径

### 10. 命名 tool profile 与受限 browser surface 已定型

当前版本把两类容易被误讲宽的能力都收口到了可验证的窄边界：

- `openyak --tool-profile <name> ...` 可以从本地 `toolProfiles` 配置中选择命名 ceiling，收窄当前 REPL 或单次 prompt 的 permission mode / allowed-tools；该能力仍然是 process-local、session transcript-only，且 sandbox 附加约束当前只覆盖 `bash`
- hidden browser tools `BrowserObserve` / `BrowserInteract` 已接入，但它们仍然要求 `browserControl.enabled=true`、显式 `--allowedTools BrowserObserve` / `BrowserInteract` 和 `danger-full-access`，同时继续保持 CLI-local、single-call、无持久 browser session、无 `/v1/threads` browser support

### 11. 新增显式 `openyak onboard` 本地 onboarding wizard

当前版本新增了一个 bounded、phase-1 的 `openyak onboard` 命令：

- 只在交互式本地终端里运行
- 不拦截 `openyak` 无参数默认进入 REPL 的行为
- 先做只读 readiness assessment
- 再按需复用 `openyak init`、用户级默认模型写入、provider-aware auth guidance / `openyak login` handoff，以及 `openyak doctor`
- 不会把 provider secrets 写入配置文件

### 12. Parity foundation 进一步落地

当前版本补上了一批更接近 parity baseline 的基础设施：

- deterministic mock Anthropic-compatible service
- mock parity harness + scenario manifest
- registry-backed Task / Team / Cron foundation tools
- tool-layer LSP / MCP registry operator surfaces
- `openyak foundations [task|team|cron|lsp|mcp]` / `/foundations [task|team|cron|lsp|mcp]` 只读发现面
- 更强的 tool-layer permission enforcement
- `openyak package-release` 生成可上传或手动分发的 release artifact 目录

这些能力的当前对外口径也已经明确冻结：

- Task / Team / Cron registry 是 `process_local_v1`，只提供进程内临时状态
- 当前版本不承诺持久化、恢复、租约或跨实例共享
- `created_at`、`updated_at`、`last_error`、`disabled_reason`、`origin`、`capabilities` 是 V1 operator contract 的核心 metadata
- `openyak server` 已提供窄化的本地 thread/session surface，但它不是 codex-style full app-server；完整独立 LSP main entry 仍然刻意后置
- `openyak foundations [task|team|cron|lsp|mcp]` 只解释当前 shipped boundary，不把 Task / Team / Cron / LSP / MCP 包装成更宽的控制面

### 13. 已新增 attach-first SDK alpha

当前版本还已经补上两个明确受边界约束的语言客户端：

- `sdk/typescript`
- `sdk/python`

它们都直接连接当前 `openyak server` 的 `/v1/threads` 合约，提供 attach-first、本地-only 的 alpha SDK，用于线程创建、运行、事件流消费与 user-input round-trip。

当前版本也明确不承诺：

- 自动拉起 server
- runtime / binary bundling
- 远端或托管部署语义
- 把旧 `/sessions` 路由作为 SDK 公共边界

### 14. 公开协作与 CI 基线已补齐

当前公开仓库现在已经具备最小协作基线：

- 根目录补齐了 `LICENSE`、`SECURITY.md`、`CODE_OF_CONDUCT.md`、`CONTRIBUTING.md`
- GitHub Actions [`.github/workflows/ci.yml`](../../../.github/workflows/ci.yml) 固化了当前已验证的 Rust、根目录 Python、Python SDK、TypeScript SDK 检查链路
- 这条 CI 基线只声明“验证当前仓库可复现的本地检查”，不宣称 release 上传、签名或自动分发已经完成

## 安装与运行

当前版本主要以源码构建方式使用：

```bash
cargo install --path crates/openyak-cli --locked
# 或
cargo build --release -p openyak-cli
```

正式 release artifact 的最小维护流程现在是：

```bash
cargo build --release -p openyak-cli
./target/release/openyak package-release --output-dir dist
```

Windows PowerShell:

```powershell
.\target\release\openyak.exe package-release --output-dir dist
```

这会生成形如 `dist/openyak-0.1.0-<target>/` 的目录，其中包含：

- `openyak` / `openyak.exe`
- `INSTALL.txt`
- `release-metadata.json`

如果 `--binary` 已经指向目标输出目录中的现成 artifact，`openyak package-release` 会显式拒绝 self-target packaging，而不是在 Windows 上退化成复制自身时的权限错误。

用户侧的 packaged-use 路径是：下载并解压该 release artifact 目录，然后直接运行其中的 `openyak` 二进制，必要时再把它放到 `PATH`。

运行示例：

```bash
openyak
openyak prompt "总结这个仓库"
openyak skills
openyak agents
openyak foundations
openyak onboard
openyak server --help
```

如果要使用 OAuth：

```bash
openyak login
```

如果要使用 GitHub PR/Issue 链路：

```bash
gh auth login --web
```

## 当前已知限制

- 当前已经支持生成 release artifact 目录，但压缩、上传、签名和自动发布流程仍未建立。
- GitHub Actions CI 目前已经覆盖公开仓库主验证基线；正式发布、上传、签名和分发流程仍待建立。
- 某些 live-provider 集成测试默认不启用，因为它们依赖真实凭据和网络。
- `openyak server` 当前只提供本地 thread/session HTTP/SSE 路由；完整独立 LSP main entry 与更宽的 server/control-plane 语义仍是后续阶段工作。
- `0.x` 阶段的命令面和交互细节仍可能继续演进。

## 建议的发布叙事

建议将 `0.1.0` 对外描述为：

> openyak 的首个 Rust 主实现版本，适合愿意从源码构建、关注本地 coding-agent CLI 与 harness engineering 的早期使用者和维护者；其 parity foundation 已具备可观测的 operator-facing baseline，并已提供窄化的 `openyak server` 本地 thread/session surface，而完整独立 LSP main entry 仍在后续里程碑中。
